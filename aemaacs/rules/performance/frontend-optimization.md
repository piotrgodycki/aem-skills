## Frontend Performance Optimization

Practical, AEM-specific techniques for optimizing frontend performance. These patterns address the unique challenges of AEM's ClientLib system, Dynamic Media integration, and Cloud Service CDN architecture.

---

### 1. Critical CSS Extraction and Inline Delivery

AEM does not provide built-in critical CSS extraction. You must implement it in the `ui.frontend` build pipeline.

**Correct -- inline critical CSS, defer the rest:**

```html
<!-- customheaderlibs.html -->
<style>
    /* Inline critical CSS for above-fold content */
    /* Generated by critters or critical npm packages during build */
    .cmp-page__header { display: flex; height: 80px; }
    .cmp-title__text { font-size: 2rem; line-height: 1.2; }
    .cmp-image__image { width: 100%; height: auto; }
    /* ... only above-fold styles, typically 10-15KB */
</style>

<!-- Defer full stylesheet -->
<link rel="preload" href="/etc.clientlibs/myproject/clientlibs/clientlib-site/resources/css/site.css"
      as="style" onload="this.onload=null;this.rel='stylesheet'" />
<noscript>
    <link rel="stylesheet" href="/etc.clientlibs/myproject/clientlibs/clientlib-site/resources/css/site.css" />
</noscript>
```

**Webpack configuration for critical CSS extraction:**

```javascript
// ui.frontend/webpack.prod.js
const CriticalCssPlugin = require('critters-webpack-plugin');

module.exports = {
    plugins: [
        new CriticalCssPlugin({
            inlineThreshold: 0,       // Inline all critical CSS
            minimumExternalSize: 0,
            pruneSource: false,        // Keep full CSS file too
            preload: 'swap',           // Preload remaining CSS
            fonts: true,               // Include critical @font-face
        })
    ]
};
```

**Incorrect -- loading all CSS as render-blocking:**
```html
<!-- This blocks rendering until entire stylesheet downloads -->
<link rel="stylesheet" href="/etc.clientlibs/myproject/clientlibs/clientlib-site.css" />
<!-- If the CSS file is 150KB+, LCP suffers significantly -->
```

### 2. JavaScript Defer/Async Loading Patterns

AEM ClientLibs load synchronously by default. You must explicitly configure deferred loading.

#### Split CSS and JS Loading

```html
<!-- customheaderlibs.html: CSS only (must be render-blocking) -->
<sly data-sly-use.clientlib="/libs/granite/sightly/templates/clientlib.html">
    <sly data-sly-call="${clientlib.css @ categories='myproject.base'}" />
</sly>

<!-- customfooterlibs.html: JS deferred at end of body -->
<sly data-sly-use.clientlib="/libs/granite/sightly/templates/clientlib.html">
    <sly data-sly-call="${clientlib.js @ categories='myproject.base'}" />
</sly>
```

#### Async/Defer for Custom Scripts

AEM's built-in ClientLib template does not support `async` or `defer` attributes. Use custom script tags or a community solution:

```html
<!-- customfooterlibs.html: manually add defer attribute -->
<script defer src="/etc.clientlibs/myproject/clientlibs/clientlib-site.js"></script>

<!-- Or use the aem-clientlib-async pattern -->
<sly data-sly-use.clientlib="/apps/myproject/components/clientlib-async/clientlib-async.html">
    <sly data-sly-call="${clientlib.js @ categories='myproject.base', loading='defer'}" />
</sly>
```

#### Module/NoModule Pattern for Modern Browsers

```html
<!-- Serve ES modules to modern browsers, fallback for legacy -->
<script type="module" src="/etc.clientlibs/myproject/clientlibs/clientlib-site.esm.js"></script>
<script nomodule src="/etc.clientlibs/myproject/clientlibs/clientlib-site.legacy.js"></script>
```

**Correct -- categorize scripts by loading priority:**
```
Eager (blocking):    Critical layout JS (rare -- avoid if possible)
Defer:               Main site JS, component initialization
Async:               Analytics, tag managers, non-critical features
Lazy (on demand):    Below-fold components, modals, chat widgets
```

**Incorrect -- loading all JS synchronously in head:**
```html
<sly data-sly-call="${clientlib.all @ categories='myproject.base'}" />
<!-- Blocks parsing AND rendering. Massive INP and LCP impact. -->
```

### 3. Image Optimization

#### Lazy Loading

AEM Core Components Image (v2+) supports native lazy loading:

```html
<!-- Core Components Image output with lazy loading -->
<div class="cmp-image" data-cmp-is="image">
    <img class="cmp-image__image"
         src="/content/dam/mysite/hero.jpg/_jcr_content/renditions/cq5dam.web.1280.1280.jpeg"
         srcset="..."
         sizes="(min-width: 1200px) 1200px, 100vw"
         alt="Hero image"
         loading="lazy"
         decoding="async" />
</div>
```

**Correct -- eager load above-fold, lazy load below-fold:**
```html
<!-- Hero/LCP image: eager load (default for first image) -->
<img src="hero.jpg" loading="eager" fetchpriority="high" />

<!-- Below-fold images: lazy load -->
<img src="card.jpg" loading="lazy" decoding="async" />
```

**Incorrect -- lazy loading the LCP image:**
```html
<!-- This DELAYS your largest contentful paint -->
<img src="hero.jpg" loading="lazy" />
<!-- The browser won't start loading until the image enters the viewport -->
```

#### Web-Optimized Image Delivery (AEM Cloud Service)

AEM as a Cloud Service automatically delivers images in WebP format (approximately 25% smaller), with transparent fallback to JPEG/PNG for unsupported browsers.

**Enable in template policy:**
1. Edit page template > Image component policy
2. Enable "Web Optimized Images"
3. No frontend code changes needed -- same `<img>` markup, different server response

**Custom component integration:**

```java
// Use AssetDelivery API for web-optimized URLs in custom components
@OSGiService
private AssetDelivery assetDelivery;

public String getOptimizedImageUrl() {
    Resource imageResource = resource.getChild("image/file");
    if (imageResource != null && assetDelivery != null) {
        Map<String, Object> params = Map.of("width", 1200, "format", "webp");
        return assetDelivery.getDeliveryURL(imageResource, params);
    }
    return null;
}
```

**Important:** Web-optimized delivery is NOT available on local SDK or AEM 6.5. The system falls back to the Adaptive Image Servlet transparently -- no markup changes needed.

#### Dynamic Media Responsive Images

```html
<!-- Dynamic Media with srcset for responsive delivery -->
<picture>
    <source media="(min-width: 1200px)"
            srcset="https://images.mysite.com/is/image/myco/hero?wid=1920&fmt=webp"
            type="image/webp" />
    <source media="(min-width: 768px)"
            srcset="https://images.mysite.com/is/image/myco/hero?wid=1200&fmt=webp"
            type="image/webp" />
    <source media="(min-width: 768px)"
            srcset="https://images.mysite.com/is/image/myco/hero?wid=1200&fmt=jpeg" />
    <img src="https://images.mysite.com/is/image/myco/hero?wid=800&fmt=jpeg"
         alt="Hero image"
         loading="eager"
         fetchpriority="high"
         width="1920" height="1080" />
</picture>
```

**Always include `width` and `height` attributes** to prevent CLS (Cumulative Layout Shift). The browser uses these to calculate aspect ratio before the image loads.

### 4. Font Optimization

#### Preload and WOFF2

```html
<!-- customheaderlibs.html: preload critical fonts -->
<link rel="preload"
      href="/etc.clientlibs/myproject/clientlibs/clientlib-base/resources/fonts/brand-regular.woff2"
      as="font" type="font/woff2" crossorigin="anonymous" />
<link rel="preload"
      href="/etc.clientlibs/myproject/clientlibs/clientlib-base/resources/fonts/brand-bold.woff2"
      as="font" type="font/woff2" crossorigin="anonymous" />
```

#### font-display Strategy

```css
@font-face {
    font-family: 'BrandFont';
    src: url('../resources/fonts/brand-regular.woff2') format('woff2');
    font-weight: 400;
    font-style: normal;
    font-display: swap; /* Show fallback immediately, swap when loaded */
}

@font-face {
    font-family: 'BrandFont';
    src: url('../resources/fonts/brand-bold.woff2') format('woff2');
    font-weight: 700;
    font-style: normal;
    font-display: swap;
}
```

**font-display values and when to use them:**

| Value | Behavior | Use When |
|-------|----------|----------|
| `swap` | Show fallback, swap when ready | Body text, headings -- always show content |
| `optional` | Show fallback, swap only if font loads in ~100ms | Non-critical decorative fonts |
| `fallback` | Brief blank, then fallback, then swap | Balance between swap and block |
| `block` | Hide text until font loads (up to 3s) | Icon fonts only (never for text) |

#### Font Subsetting

Reduce font file sizes by including only the characters you need:

```bash
# Using pyftsubset to create a Latin-only subset
pip install fonttools brotli
pyftsubset BrandFont.ttf \
    --output-file=BrandFont-latin.woff2 \
    --flavor=woff2 \
    --layout-features='kern,liga' \
    --unicodes='U+0000-00FF,U+0131,U+0152-0153,U+02BB-02BC,U+2000-206F'
```

**Correct -- WOFF2 with preload and swap:**
```css
/* WOFF2 saves ~30% vs WOFF, ~50% vs TTF */
@font-face {
    font-family: 'Brand';
    src: url('brand.woff2') format('woff2');
    font-display: swap;
}
```

**Incorrect -- loading TTF/OTF directly:**
```css
/* TTF files are 2-3x larger than WOFF2 */
@font-face {
    font-family: 'Brand';
    src: url('brand.ttf') format('truetype');
    /* Missing font-display causes invisible text during load */
}
```

### 5. Service Worker Patterns for AEM

Service workers are useful for offline support and aggressive caching but require careful handling with AEM's content authoring model.

```javascript
// service-worker.js
const CACHE_NAME = 'mysite-v1';
const PRECACHE_URLS = [
    '/etc.clientlibs/myproject/clientlibs/clientlib-site.css',
    '/etc.clientlibs/myproject/clientlibs/clientlib-site.js',
    '/etc.clientlibs/myproject/clientlibs/clientlib-base/resources/fonts/brand.woff2'
];

self.addEventListener('install', event => {
    event.waitUntil(
        caches.open(CACHE_NAME)
            .then(cache => cache.addAll(PRECACHE_URLS))
    );
});

self.addEventListener('fetch', event => {
    const url = new URL(event.request.url);

    // Cache-first for static assets (ClientLibs, fonts, icons)
    if (url.pathname.startsWith('/etc.clientlibs/')) {
        event.respondWith(
            caches.match(event.request).then(cached =>
                cached || fetch(event.request).then(response => {
                    const clone = response.clone();
                    caches.open(CACHE_NAME).then(cache => cache.put(event.request, clone));
                    return response;
                })
            )
        );
        return;
    }

    // Network-first for content pages (content changes frequently)
    if (url.pathname.startsWith('/content/')) {
        event.respondWith(
            fetch(event.request)
                .then(response => {
                    const clone = response.clone();
                    caches.open(CACHE_NAME).then(cache => cache.put(event.request, clone));
                    return response;
                })
                .catch(() => caches.match(event.request))
        );
        return;
    }
});
```

**Important AEM-specific considerations:**
- Do NOT cache on author instances (check `wcmmode` before registering)
- Bust service worker cache on deployment (version the cache name)
- AEM's CDN (Fastly) already provides edge caching -- service workers add browser-level caching

```javascript
// Only register service worker on publish
if ('serviceWorker' in navigator && !document.cookie.includes('wcmmode=')) {
    navigator.serviceWorker.register('/sw.js');
}
```

### 6. Resource Hints: Preconnect, Prefetch, and Preload

```html
<!-- customheaderlibs.html -->

<!-- Preconnect: establish early connections to critical third-party origins -->
<link rel="preconnect" href="https://fonts.googleapis.com" />
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin />
<link rel="preconnect" href="https://images.mysite.com" />  <!-- Dynamic Media -->
<link rel="preconnect" href="https://www.googletagmanager.com" />

<!-- DNS-Prefetch: fallback for browsers without preconnect support -->
<link rel="dns-prefetch" href="https://fonts.googleapis.com" />
<link rel="dns-prefetch" href="https://www.google-analytics.com" />

<!-- Preload: critical resources that the browser discovers late -->
<link rel="preload" href="/etc.clientlibs/myproject/clientlibs/clientlib-base/resources/fonts/brand.woff2"
      as="font" type="font/woff2" crossorigin />
<link rel="preload" as="image" href="/content/dam/mysite/hero.jpg/_jcr_content/renditions/cq5dam.web.1280.1280.webp"
      type="image/webp" />

<!-- Prefetch: resources likely needed on next navigation -->
<link rel="prefetch" href="/content/mysite/en/about.html" />
```

**Correct -- preconnect only to origins you will use immediately:**
```html
<link rel="preconnect" href="https://images.mysite.com" />
<!-- Used for hero image that loads immediately -->
```

**Incorrect -- preconnecting to everything:**
```html
<!-- Each preconnect costs a TCP+TLS handshake -->
<!-- Only preconnect to origins used in the first ~5 seconds -->
<link rel="preconnect" href="https://chat-widget.example.com" />
<!-- Chat loads after 10 seconds -- use dns-prefetch instead -->
```

#### Webpack Bundle Analyzer

```javascript

module.exports = {

            openAnalyzer: false,
            defaultSizes: 'gzip'             // Show gzipped sizes
        })
    ]
};
```

#### Code Splitting for AEM Components

```javascript
// Lazy-load component JS only when the component exists on the page
function initComponents() {
    // Eager: always loaded
    import('./components/navigation/navigation').then(m => m.init());

    // Lazy: loaded only if component is present in DOM
    if (document.querySelector('[data-cmp-is="carousel"]')) {
        import(/* webpackChunkName: "carousel" */ './components/carousel/carousel')
            .then(m => m.init());
    }

    if (document.querySelector('[data-cmp-is="accordion"]')) {
        import(/* webpackChunkName: "accordion" */ './components/accordion/accordion')
            .then(m => m.init());
    }

    // Very lazy: loaded on user interaction
    document.querySelector('.search-trigger')?.addEventListener('click', () => {
        import(/* webpackChunkName: "search" */ './components/search/search')
            .then(m => m.openSearch());
    }, { once: true });
}
```

#### Intersection Observer for Below-Fold Components

```javascript
// Load component JS when it scrolls into view
const observer = new IntersectionObserver((entries) => {
    entries.forEach(entry => {
        if (entry.isIntersecting) {
            const component = entry.target.dataset.cmpIs;
            import(`./components/${component}/${component}.js`)
                .then(m => m.init(entry.target));
            observer.unobserve(entry.target);
        }
    });
}, { rootMargin: '200px' }); // Start loading 200px before visible

document.querySelectorAll('[data-cmp-lazy]').forEach(el => observer.observe(el));
```

### 8. Tree Shaking Configuration

Ensure dead code elimination works in the `ui.frontend` webpack build:

```javascript
// ui.frontend/webpack.prod.js
module.exports = {
    mode: 'production',
    optimization: {
        usedExports: true,     // Mark unused exports
        sideEffects: true,     // Respect package.json sideEffects
        minimize: true,
        minimizer: [
            new TerserPlugin({
                terserOptions: {
                    compress: { drop_console: true },  // Remove console.log
                    output: { comments: false }
                }
            })
        ]
    }
};
```

```json
// ui.frontend/package.json
{
    "sideEffects": [
        "*.css",
        "*.scss",
        "**/*.css"
    ]
}
```

**Correct -- use named exports for tree-shakeable modules:**
```javascript
// utils.js
export function formatDate(date) { ... }
export function formatCurrency(amount) { ... }
// Only formatDate is imported? formatCurrency is removed from bundle.
```

**Incorrect -- default exports prevent tree shaking:**
```javascript
// utils.js
export default {
    formatDate(date) { ... },
    formatCurrency(amount) { ... }
};
// Entire object is included even if only formatDate is used.
```

### 9. Core Web Vitals Optimization in AEM

#### Largest Contentful Paint (LCP)

Target: under 2.5 seconds.

**AEM-specific LCP optimizations:**

```html
<!-- 1. Preload the LCP image -->
<link rel="preload" as="image"
      href="/content/dam/mysite/hero.webp"
      fetchpriority="high" />

<!-- 2. Use fetchpriority on the LCP image element -->
<img src="/content/dam/mysite/hero.webp"
     fetchpriority="high"
     loading="eager"
     decoding="sync"
     width="1920" height="1080"
     alt="Hero" />

<!-- 3. Inline critical CSS to avoid render-blocking -->
<!-- 4. Ensure TTFB is fast (AEM CDN/Fastly handles this) -->
```

#### Interaction to Next Paint (INP)

Target: under 200 milliseconds.

```javascript
// Break long tasks into smaller chunks
function initAllComponents() {
    const components = document.querySelectorAll('[data-cmp-is]');

    // INCORRECT: Initialize all at once (blocks main thread)
    // components.forEach(el => initComponent(el));

    // CORRECT: Yield to main thread between initializations
    const queue = [...components];
    function processNext() {
        if (queue.length === 0) return;
        const el = queue.shift();
        initComponent(el);
        // Yield to browser -- allows handling user input between inits
        setTimeout(processNext, 0);
    }
    processNext();
}

// Use requestIdleCallback for non-critical work
requestIdleCallback(() => {
    initAnalyticsMutationObserver();
    prefetchNextPageLinks();
});
```

#### Cumulative Layout Shift (CLS)

Target: under 0.1.

```css
/* 1. Always set explicit dimensions on images */
.cmp-image__image {
    width: 100%;
    height: auto;
    aspect-ratio: 16 / 9;  /* Prevents layout shift */
}

/* 2. Reserve space for dynamic content */
.cmp-embed__wrapper {
    aspect-ratio: 16 / 9;
    background: #f0f0f0;  /* Placeholder color */
}

/* 3. Reserve space for fonts (match fallback metrics) */
@font-face {
    font-family: 'BrandFont';
    src: url('brand.woff2') format('woff2');
    font-display: swap;
    /* Use size-adjust to match fallback font metrics */
    size-adjust: 105%;
    ascent-override: 95%;
}

/* 4. Prevent ad/embed slots from shifting content */
.ad-slot {
    min-height: 250px;  /* Reserve space for standard ad unit */
}
```

**AEM-specific CLS pitfall -- the responsive grid:**
```css
/* AEM's responsive grid classes can cause CLS if CSS loads late */
/* Ensure grid CSS is in critical CSS or loads early */
.aem-Grid {
    display: flex;
    flex-wrap: wrap;
}
```

### 10. Lighthouse Audit Patterns for AEM Sites

#### Common AEM-Specific Lighthouse Issues and Fixes

| Issue | Cause | Fix |
|-------|-------|-----|
| Render-blocking resources | ClientLibs loaded with `clientlib.all` in head | Split CSS/JS: CSS in head, JS deferred in footer |
| Unused CSS | Full Bootstrap or grid framework loaded | Tree shake, load component CSS only when needed |
| Images without dimensions | Core Component image missing width/height | Ensure image component policy includes dimensions |
| Missing preconnect | Dynamic Media, fonts, GTM origins | Add preconnect hints in customheaderlibs.html |
| Large JavaScript bundles | No code splitting in ui.frontend | Dynamic imports for below-fold components |
| Text invisible during load | Missing font-display on @font-face | Add `font-display: swap` to all font declarations |
| Excessive DOM size | Deeply nested AEM containers | Flatten component structure, limit nesting depth |

### 11. Third-Party Script Management

#### GTM, Analytics, and Chat Widgets

```html
<!-- customfooterlibs.html: load third-party scripts only on publish -->
<sly data-sly-test="${wcmmode.disabled}">

    <!-- GTM: load async, never defer (GTM manages its own timing) -->
    <script async src="https://www.googletagmanager.com/gtm.js?id=${properties.gtmId || 'GTM-XXXX'}"></script>

    <!-- Data Layer: must be before GTM -->
    <script>
        window.dataLayer = window.dataLayer || [];
        window.adobeDataLayer = window.adobeDataLayer || [];
    </script>
</sly>
```

#### Delayed Third-Party Loading Pattern

```javascript
// Load non-critical third parties after page is interactive
function loadDelayed() {
    // Chat widget: load after 5 seconds or first user interaction
    const loadChat = () => {
        if (document.querySelector('#chat-widget')) return;
        const script = document.createElement('script');
        script.src = 'https://chat.example.com/widget.js';
        script.async = true;
        document.body.appendChild(script);
    };

    // Trigger on first interaction or after timeout
    const triggers = ['mousemove', 'touchstart', 'scroll', 'keydown'];
    const handler = () => {
        loadChat();
        triggers.forEach(e => document.removeEventListener(e, handler));
    };
    triggers.forEach(e => document.addEventListener(e, handler, { once: true, passive: true }));
    setTimeout(handler, 5000);
}

// Run after main content is loaded
if (document.readyState === 'complete') {
    loadDelayed();
} else {
    window.addEventListener('load', loadDelayed);
}
```

**Correct -- phased loading approach:**
```
Phase 1 (0-1s):    Critical CSS, LCP image, core JS
Phase 2 (1-3s):    Full CSS, component JS, below-fold images
Phase 3 (3-5s):    Analytics, GTM, A/B testing
Phase 4 (5s+):     Chat, surveys, social widgets
```

**Incorrect -- loading everything at once:**
```html
<!-- All scripts in head, synchronous, on every page load -->
<script src="gtm.js"></script>
<script src="hotjar.js"></script>
<script src="chat.js"></script>
<script src="analytics.js"></script>
<!-- Total Blocking Time: catastrophic -->
```

### 13. HTTP/2 and CDN Optimization

AEM Cloud Service uses Fastly CDN with HTTP/2 enabled by default.

**HTTP/2 considerations for AEM:**

```
With HTTP/2:
  - Embedding ClientLibs is LESS important (multiplexing reduces overhead)
  - Multiple smaller files can be better than one large bundle
  - Server push is deprecated in most browsers -- use preload hints instead
  - Domain sharding is counterproductive (breaks connection reuse)
```

**Correct -- leverage HTTP/2 multiplexing:**
```xml
<!-- Fewer embeds, more granular categories -->
<jcr:root jcr:primaryType="cq:ClientLibraryFolder"
    categories="[myproject.navigation]"
    allowProxy="{Boolean}true" />
<!-- Each component can have its own small ClientLib -->
```

**Incorrect -- excessive embedding for HTTP/1.1 optimization:**
```xml
<!-- Embedding everything into one massive file is unnecessary with HTTP/2 -->
<jcr:root jcr:primaryType="cq:ClientLibraryFolder"
    categories="[myproject.base]"
    embed="[myproject.utils,myproject.nav,myproject.header,myproject.footer,
            myproject.carousel,myproject.accordion,myproject.tabs,
            myproject.search,myproject.forms]"
    allowProxy="{Boolean}true" />
<!-- One 500KB file vs many small files: HTTP/2 favors smaller files -->
```

#### Request Log Analysis

```bash

# - Number of includes (high count = too many nested components)
```

### 15. Real User Monitoring (RUM) vs Synthetic Testing

#### Lighthouse Is Not Enough

Lighthouse runs in a controlled lab environment with simulated throttling. It is useful for catching regressions and directional guidance, but it **does not reflect real user experience**:

| Aspect | Lighthouse (Synthetic) | RUM (Field Data) |
|--------|----------------------|------------------|
| Network | Simulated throttling | Real user connections (3G, 4G, WiFi, fibre) |
| Device | Emulated mobile | Real devices (low-end Android, iPhones, desktops) |
| Geography | Single location | Global distribution |
| Sample size | 1 page load | Thousands of real visits |
| SEO impact | None directly | CrUX data drives Core Web Vitals ranking signal |
| Caching | Cold cache only | Mix of warm/cold cache |

**Google uses CrUX (Chrome User Experience Report) — field data — for ranking, not Lighthouse scores.**

#### RUM Sources for AEM

1. **AEM Cloud Service Operational Telemetry** — Built-in RUM that captures CWV, page load timing, CDN hit rates
2. **Chrome User Experience Report (CrUX)** — Public dataset of real Chrome user metrics, queryable via BigQuery or PageSpeed Insights
3. **Google Search Console** — Core Web Vitals report with field data grouped by URL pattern
4. **web-vitals library** — Instrument your own RUM collection:

```javascript
// In a ClientLib loaded on all pages (non-blocking)
import { onLCP, onINP, onCLS, onTTFB } from 'web-vitals';

function sendToAnalytics(metric) {
  const body = JSON.stringify({
    name: metric.name,
    value: metric.value,
    rating: metric.rating, // "good", "needs-improvement", "poor"
    delta: metric.delta,
    id: metric.id,
    navigationType: metric.navigationType,
  });
  // Use sendBeacon for reliability during page unload
  navigator.sendBeacon('/analytics/cwv', body);
}

onLCP(sendToAnalytics);
onINP(sendToAnalytics);
onCLS(sendToAnalytics);
onTTFB(sendToAnalytics);
```

#### RUM-Driven Optimization Workflow

1. **Measure** — Collect RUM data for 7+ days to get statistically significant samples
2. **Identify** — Find the worst-performing pages/URLs by p75 CWV values
3. **Diagnose** — Use Lighthouse/DevTools on those specific pages to find root causes
4. **Fix** — Apply targeted optimizations
5. **Validate** — Wait 28 days for CrUX to update, confirm improvement in field data

**Incorrect — optimizing based on Lighthouse alone:**
```
Lighthouse score: 95 → "We're good!"
But CrUX p75 LCP: 4.2s (poor) — real users on slow connections have bad experience
```

**Correct — using RUM as source of truth:**
```
CrUX p75 LCP: 4.2s → investigate real user waterfalls
Root cause: hero image not preloaded, 800ms TTFB on CDN miss
Fix: add <link rel="preload">, tune cache headers
Validate: CrUX p75 LCP drops to 2.1s after 28 days
```

### 16. TTFB Optimization and HTTP/3

#### TTFB Target: Below 800ms

Time to First Byte (TTFB) measures the time from navigation start to when the first byte of the HTML response arrives. Google considers TTFB "good" at ≤ 800ms and "poor" at > 1800ms. For AEM Cloud Service:

```
TTFB = DNS + TCP + TLS + Request to CDN + (CDN miss? → Dispatcher → AEM Publish)
```

**Optimization strategies:**

1. **CDN cache hit rate** — The #1 TTFB factor. Maximize cache hits:
   ```apache
   # HTML pages: 5min browser, 1hr CDN, with background refresh
   <LocationMatch "^/content/.*\.(html)$">
       Header set Cache-Control "max-age=300,stale-while-revalidate=3600,public"
       Header set Surrogate-Control "max-age=3600,stale-while-revalidate=86400"
   </LocationMatch>
   ```

2. **Preconnect to AEM CDN** — Eliminate DNS+TCP+TLS latency for subdomains:
   ```html
   <!-- In head.html or customheaderlibs.html -->
   <link rel="preconnect" href="https://publish-p12345-e67890.adobeaemcloud.com" crossorigin>
   <link rel="dns-prefetch" href="https://publish-p12345-e67890.adobeaemcloud.com">
   ```

3. **Minimize server-side rendering time** — Target < 200ms AEM response:
   - Avoid slow JCR queries in Sling Models (see `query-optimization.md`)
   - Use SDI for personalized components (see `dispatcher-caching.md`)
   - Minimize nested `data-sly-resource` calls

4. **Use `stale-while-revalidate`** — Serve cached content instantly while refreshing in background

#### HTTP/3 (QUIC)

HTTP/3 uses QUIC (UDP-based) instead of TCP, providing:
- **0-RTT connection establishment** — eliminates TCP+TLS handshake latency
- **No head-of-line blocking** — multiplexed streams independent of each other
- **Better on lossy networks** — UDP handles packet loss more gracefully (critical for mobile)

AEM Cloud Service CDN (Fastly) supports HTTP/3 automatically. To verify:

```bash
# Check HTTP/3 support
curl --http3 -I https://your-site.com
# Look for: HTTP/3 200
# Or in Chrome DevTools > Network > Protocol column: "h3"
```

**Frontend implications of HTTP/3:**
- **Less need to bundle aggressively** — multiplexing makes many small requests efficient
- **Keep-alive is free** — no TCP connection overhead
- **Mobile performance improves significantly** — QUIC handles network switches (WiFi→cellular) gracefully
- **Still preload critical resources** — 0-RTT doesn't eliminate the need for resource hints

**Incorrect — assuming HTTP/2 patterns still apply:**
```
# Over-bundling everything into one massive ClientLib
# because "fewer HTTP requests = faster" — not true with HTTP/3
embed="[lib1,lib2,lib3,lib4,lib5,lib6,lib7,lib8]"
```

**Correct — leverage HTTP/3 multiplexing:**
```
# Smaller, purpose-specific ClientLibs that load in parallel
# HTTP/3 handles many concurrent requests efficiently
categories="[myproject.header]"    # 15KB
categories="[myproject.hero]"      # 8KB
categories="[myproject.base]"      # 40KB
```

### 17. `<head>` Element Order (capo.js)

The order of elements in `<head>` significantly impacts performance. [capo.js](https://github.com/rviscomi/capo.js) is a tool that analyzes and scores your `<head>` element ordering.

#### Optimal `<head>` Order

Elements should appear in this order for best performance:

```html
<head>
  <!-- 1. Meta charset (MUST be first, within first 1024 bytes) -->
  <meta charset="utf-8">

  <!-- 2. Viewport meta -->
  <meta name="viewport" content="width=device-width, initial-scale=1">

  <!-- 3. Title -->
  <title>Page Title</title>

  <!-- 4. Preconnect hints (before any resources from those origins) -->
  <link rel="preconnect" href="https://cdn.example.com" crossorigin>
  <link rel="preconnect" href="https://fonts.googleapis.com" crossorigin>

  <!-- 5. Preload critical resources (fonts, hero image, critical JS) -->
  <link rel="preload" href="/fonts/brand.woff2" as="font" type="font/woff2" crossorigin>
  <link rel="preload" as="image" href="/content/dam/hero.jpg" fetchpriority="high">

  <!-- 6. Synchronous CSS (render-blocking — only critical styles) -->
  <sly data-sly-call="${clientLib.css @ categories='myproject.critical'}"/>

  <!-- 7. Synchronous JS (if absolutely necessary — avoid) -->

  <!-- 8. Deferred/async JS -->
  <script src="/etc.clientlibs/myproject/clientlibs/clientlib-site.js" defer></script>

  <!-- 9. Prefetch/prerender hints (for likely next navigation) -->
  <link rel="prefetch" href="/next-page.html">

  <!-- 10. SEO and social meta tags -->
  <meta name="description" content="...">
  <meta property="og:title" content="...">
  <link rel="canonical" href="...">

  <!-- 11. Deferred CSS (non-critical styles) -->
  <link rel="stylesheet" href="non-critical.css" media="print" onload="this.media='all'">

  <!-- 12. Favicon -->
  <link rel="icon" href="/favicon.ico">
</head>
```

#### Why Order Matters

- **Meta charset first**: Browsers may restart parsing if charset is found late
- **Preconnect before resources**: DNS+TCP+TLS starts before the resource is requested
- **CSS before JS**: CSS is render-blocking; getting it earlier means faster First Paint
- **Deferred JS after CSS**: Prevents JS from blocking CSS download/parsing
- **SEO meta late**: Does not affect rendering — put it after performance-critical elements

#### AEM Implementation in `customheaderlibs.html`

```html
<!--/* customheaderlibs.html — loaded inside <head> by page component */-->

<!-- Preconnect to CDN origins -->
<link rel="preconnect" href="https://assets.adobedtm.com" crossorigin>

<!-- Preload critical font -->
<link rel="preload" href="${clientLib.path @ categories='myproject.base'}/resources/brand-regular.woff2"
      as="font" type="font/woff2" crossorigin>

<!-- Critical CSS -->
<sly data-sly-call="${clientLib.css @ categories='myproject.base'}"/>

<!-- Deferred site JS -->
<sly data-sly-call="${clientLib.js @ categories='myproject.base'}"/>
```

#### Testing with capo.js

```bash
# Install capo.js Chrome extension or use CLI
npx capo.js https://your-aem-site.com

# Score: each misplaced element gets a penalty
# Target: minimal warnings, all critical resources in correct position
```

### 18. Cookie Consent Loading Best Practices

Cookie consent banners are the #1 third-party performance killer when loaded incorrectly. **The best approach is to load consent management through Google Tag Manager (GTM) in deferred mode.**

#### Why GTM for Cookie Consent

1. **Single script entry point** — GTM loads asynchronously, consent banner loads within GTM
2. **Consent-aware tag firing** — GTM's built-in consent mode (`gtag('consent', ...)`) gates all other tags
3. **No render-blocking** — consent UI appears after page content is interactive
4. **Centralized management** — marketing team manages consent config in GTM without code deploys

#### Implementation Pattern

**Step 1: Load GTM deferred (not in `<head>`)**

```html
<!--/* customfooterlibs.html — at end of body */-->

<!-- GTM with consent defaults -->
<script>
  // Set consent defaults BEFORE GTM loads
  window.dataLayer = window.dataLayer || [];
  function gtag(){dataLayer.push(arguments);}

  // Default: deny all until consent is given
  gtag('consent', 'default', {
    'analytics_storage': 'denied',
    'ad_storage': 'denied',
    'ad_user_data': 'denied',
    'ad_personalization': 'denied',
    'functionality_storage': 'denied',
    'personalization_storage': 'denied',
    'security_storage': 'granted',
    'wait_for_update': 500
  });
</script>

<!-- GTM script — async, non-blocking -->
<script async src="https://www.googletagmanager.com/gtm.js?id=GTM-XXXX"></script>
```

**Step 2: Consent banner triggers in GTM**

Configure your CMP (OneTrust, Cookiebot, Usercentrics, etc.) as a GTM tag:
- **Trigger**: Page View — DOM Ready (not Page View — Initialization)
- **Tag type**: Custom HTML with the CMP script
- The CMP calls `gtag('consent', 'update', {...})` when user makes a choice

**Step 3: All tracking tags gated by consent**

In GTM, set all analytics/marketing tags to require consent:
- Google Analytics: requires `analytics_storage: 'granted'`
- Facebook Pixel: requires `ad_storage: 'granted'`
- etc.

#### Performance Impact

```
❌ BAD: Consent banner in <head> (render-blocking)
   → +500-1500ms to LCP, visible layout shift

❌ BAD: Consent banner as synchronous script before content
   → Blocks entire page rendering until CMP JS downloads + executes

✅ GOOD: GTM async in footer, CMP as GTM tag firing on DOM Ready
   → Zero impact on LCP, minimal impact on INP
   → Consent UI appears ~1-2s after page load (acceptable)
```

#### AEM-Specific Pattern

```javascript
// ClientLib JS: check consent state before initializing features
function isConsentGranted(purpose) {
  // Check GTM consent state
  const consentState = window.google_tag_data?.ics?.entries;
  return consentState?.[purpose]?.granted === true;
}

// Only initialize analytics if consent is granted
if (isConsentGranted('analytics_storage')) {
  initializeAnalytics();
}

// Listen for consent updates
window.addEventListener('consent-update', function(e) {
  if (e.detail.analytics_storage === 'granted') {
    initializeAnalytics();
  }
});
```

#### Anti-Patterns

- Loading CMP script synchronously in `<head>` — blocks rendering
- Multiple consent scripts (CMP + GTM + analytics each checking consent separately)
- Consent wall that prevents any content rendering — use a non-blocking banner
- Not setting `consent: 'default'` before GTM loads — tags fire briefly before being blocked
- Cookie consent CSS in main ClientLib — load it with the CMP

### 19. Speculation Rules API

Speculation Rules allow the browser to prefetch or prerender likely next navigations, making them feel instant:

```html
<!--/* customfooterlibs.html — add speculation rules */-->
<script type="speculationrules">
{
  "prerender": [{
    "where": {
      "and": [
        { "href_matches": "/content/mysite/*" },
        { "not": { "href_matches": "*/logout*" } },
        { "not": { "selector_matches": "[data-no-prerender]" } }
      ]
    },
    "eagerness": "moderate"
  }],
  "prefetch": [{
    "where": { "href_matches": "/content/mysite/*" },
    "eagerness": "moderate"
  }]
}
</script>
```

**Eagerness levels:**
- `immediate` — speculate as soon as rules are observed
- `eager` — earlier than moderate
- `moderate` — on hover (200ms) — **recommended for most AEM sites**
- `conservative` — on pointer down (click start)

AEM-specific considerations:
- Exclude authenticated pages (`/content/*/user/*`)
- Exclude pages with side effects (forms, checkout)
- Use `moderate` eagerness to avoid prerendering on every hover
- Works with AEM's vanity URLs and Sling Mappings
- Check `HTMLScriptElement.supports?.('speculationrules')` before injecting dynamically

### 20. Back/Forward Cache (bfcache)

bfcache stores a complete page snapshot when the user navigates away. Back/Forward restores it instantly (0ms load).

**Patterns that break bfcache (avoid these):**

```javascript
// ❌ unload event prevents bfcache
window.addEventListener('unload', () => { ... });

// ❌ Cache-Control: no-store prevents bfcache
// (don't set this on cacheable HTML pages)
```

**Correct — bfcache-friendly:**

```javascript
// ✅ Use pagehide instead of unload
window.addEventListener('pagehide', (event) => {
  if (event.persisted) { /* going into bfcache */ }
});

// ✅ Refresh dynamic content on bfcache restore
window.addEventListener('pageshow', (event) => {
  if (event.persisted) {
    // Restored from bfcache — update stale content
    document.querySelectorAll('[data-dynamic]').forEach(el => {
      refreshComponent(el);
    });
  }
});
```

**Test bfcache:** Chrome DevTools > Application > Back/forward cache > "Test back/forward cache"

AEM-specific bfcache blockers to watch:
- ContextHub using `unload` event listeners
- Analytics scripts (Adobe Launch/at.js) with `unload` handlers
- ActiveMQ/websocket connections for author-side features leaking to publish

## Project Lombok Best Practices in AEM Cloud Service

Project Lombok eliminates repetitive Java boilerplate (getters, setters, constructors, logging, toString) through compile-time annotation processing. In AEM projects it is especially valuable for Sling Models (dozens of getter methods for HTL), OSGi services (logging), and JSON DTOs (Sling Model Exporters). However, several Lombok annotations conflict with AEM's injection framework and must be used carefully.

---

### 1. Lombok in AEM Context

#### Why Lombok in AEM

AEM backend code is extremely boilerplate-heavy. A typical Sling Model with 15 dialog fields requires 15 getter methods just for HTL access. OSGi services need a logger field in every class. Lombok eliminates all of this while keeping the compiled bytecode identical to hand-written code.

#### Maven Dependency Configuration

Add Lombok with `provided` scope — it is only needed at compile time and must not be bundled into the OSGi bundle:

```xml
<!-- In core/pom.xml (the bundle module) -->
<dependency>
    <groupId>org.projectlombok</groupId>
    <artifactId>lombok</artifactId>
    <version>1.18.34</version>
    <scope>provided</scope>
</dependency>
```

Configure the `maven-compiler-plugin` to include Lombok's annotation processor:

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <configuration>
        <source>11</source>
        <target>11</target>
        <annotationProcessorPaths>
            <path>
                <groupId>org.projectlombok</groupId>
                <artifactId>lombok</artifactId>
                <version>1.18.34</version>
            </path>
        </annotationProcessorPaths>
    </configuration>
</plugin>
```

Optionally add the `lombok-maven-plugin` for delombok (generating source with Lombok annotations expanded — useful for Javadoc and debugging):

```xml
<plugin>
    <groupId>org.projectlombok</groupId>
    <artifactId>lombok-maven-plugin</artifactId>
    <version>1.18.20.0</version>
    <executions>
        <execution>
            <phase>generate-sources</phase>
            <goals>
                <goal>delombok</goal>
            </goals>
        </execution>
    </executions>
</plugin>
```

#### IDE Setup

| IDE | Setup |
|-----|-------|
| IntelliJ IDEA | Install the **Lombok** plugin (bundled since 2020.3). Enable `Settings > Build > Compiler > Annotation Processors > Enable annotation processing`. |
| VS Code | Install the **Lombok Annotations Support for VS Code** extension. The Java extension pack must also be installed. |
| Eclipse | Run `lombok.jar` as a Java application to install the Eclipse agent, then restart. |

#### Cloud Service Compatibility

Lombok works with AEM as a Cloud Service without restrictions. The annotation processing happens at compile time during the Cloud Manager build. The generated bytecode is standard Java — no Lombok runtime dependency is deployed to AEM. The `provided` scope ensures Lombok is never included in the OSGi bundle.

### 2. Essential Annotations for AEM

#### @Getter / @Setter — HTL Field Access

`@Getter` is the most commonly used Lombok annotation in AEM. HTL accesses Sling Model data through getter methods. Lombok generates them at compile time.

```java
// CORRECT: @Getter on individual fields for HTL access
@Model(adaptables = Resource.class,
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class HeroComponentImpl implements HeroComponent {

    @Getter
    @ValueMapValue
    private String title;

    @Getter
    @ValueMapValue
    private String description;

    @Getter
    @ValueMapValue
    private String imagePath;
}
```

```java
// CORRECT: Class-level @Getter when ALL fields should be exposed
@Getter
@Model(adaptables = Resource.class,
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class CardComponentImpl implements CardComponent {

    @ValueMapValue
    private String heading;

    @ValueMapValue
    private String subheading;

    @ValueMapValue
    private String linkUrl;
}
```

`@Setter` is rarely needed in Sling Models (fields are injected, not set externally) but is useful for test helpers and DTOs.

#### @Slf4j / @Log — Logging

`@Slf4j` generates a `private static final org.slf4j.Logger log` field. This is the correct logging API for AEM (SLF4J over Logback).

```java
// CORRECT: @Slf4j on any AEM backend class
@Slf4j
@Model(adaptables = SlingHttpServletRequest.class,
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class SearchResultsModelImpl implements SearchResultsModel {

    @PostConstruct
    private void init() {
        log.debug("Initializing search results model");
    }
}
```

```java
// CORRECT: @Slf4j on OSGi service
@Slf4j
@Component(service = AssetService.class)
public class AssetServiceImpl implements AssetService {

    @Override
    public Asset getAsset(ResourceResolver resolver, String path) {
        log.trace("Fetching asset at path: {}", path);
        // ...
    }
}
```

```java
// INCORRECT: Using @Log (generates java.util.logging.Logger — wrong API for AEM)
@Log
@Component(service = MyService.class)
public class MyServiceImpl implements MyService {
    // log.info("...") — this uses JUL, not SLF4J. AEM's Logback will not capture it correctly.
}
```

**Always use `@Slf4j`**, never `@Log`, `@Log4j`, or `@Log4j2`. AEM Cloud Service uses SLF4J backed by Logback.

#### @ToString — Debugging Components

```java
// CORRECT: @ToString for debugging, excluding JCR-specific fields
@ToString(exclude = {"resource", "request"})
@Model(adaptables = SlingHttpServletRequest.class,
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class BreadcrumbItemImpl implements BreadcrumbItem {

    @Getter
    @ValueMapValue
    private String title;

    @Getter
    @ValueMapValue
    private String linkUrl;

    @Self
    private SlingHttpServletRequest request;

    @SlingObject
    private Resource resource;
}
```

```java
// INCORRECT: @ToString including Resource — causes serialization issues and huge log output
@ToString
@Model(adaptables = Resource.class,
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class BreadcrumbItemImpl implements BreadcrumbItem {

    @SlingObject
    private Resource resource;  // toString on Resource dumps the entire subtree
}
```

#### @EqualsAndHashCode — Value Objects

Use on DTOs and value objects, never on JCR-backed objects:

```java
// CORRECT: @EqualsAndHashCode on a pure value object
@EqualsAndHashCode
@Getter
@AllArgsConstructor
public class NavigationItem {
    private final String title;
    private final String path;
    private final int level;
}
```

```java
// INCORRECT: @EqualsAndHashCode on a Sling Model backed by a Resource
@EqualsAndHashCode
@Model(adaptables = Resource.class)
public class PageCardImpl implements PageCard {
    @SlingObject
    private Resource resource;  // Resource equality is identity-based, not value-based
}
```

#### @Builder — Complex Object Construction

Excellent for navigation structures, menu items, and response DTOs:

```java
// CORRECT: @Builder on a DTO used to construct navigation trees
@Builder
@Getter
public class NavItem {
    private final String title;
    private final String url;
    private final boolean active;
    @Builder.Default
    private final List<NavItem> children = new ArrayList<>();
    @Builder.Default
    private final String target = "_self";
}

// Usage:
NavItem item = NavItem.builder()
    .title("Products")
    .url("/content/mysite/products")
    .active(true)
    .children(childItems)
    .build();
```

#### @Data — DTOs Only

`@Data` combines `@Getter`, `@Setter`, `@ToString`, `@EqualsAndHashCode`, and `@RequiredArgsConstructor`. It is appropriate for simple DTOs but **dangerous on Sling Models**.

```java
// CORRECT: @Data on a JSON response DTO
@Data
public class SearchResult {
    private String title;
    private String path;
    private String excerpt;
    private double score;
}
```

```java
// INCORRECT: @Data on a Sling Model — generates constructors that conflict with Sling injection
@Data
@Model(adaptables = Resource.class,
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class ArticleModelImpl implements ArticleModel {

    @ValueMapValue
    private String title;  // @Data generates a required-args constructor and setters — Sling
                           // injection uses no-arg construction + field injection, so this breaks
}
```

#### @Value — Immutable Value Objects

`@Value` makes all fields `private final`, generates getters (no setters), all-args constructor, `toString`, `equals`, and `hashCode`. Ideal for immutable domain objects:

```java
// CORRECT: @Value on an immutable value object
@Value
public class AssetMetadata {
    String mimeType;
    long fileSize;
    int width;
    int height;
    String title;
}
```

**Never use `@Value` on Sling Models** — the final fields and required-args constructor conflict with Sling injection.

#### @NoArgsConstructor / @AllArgsConstructor / @RequiredArgsConstructor

```java
// CORRECT: @NoArgsConstructor on a DTO that needs default construction (e.g., Jackson deserialization)
@NoArgsConstructor
@AllArgsConstructor
@Getter
@Setter
public class ApiResponse {
    private int status;
    private String message;
    private Object data;
}
```

```java
// INCORRECT: @AllArgsConstructor on Sling Model — Sling cannot use parameterized constructors for injection
@AllArgsConstructor
@Model(adaptables = Resource.class)
public class TeaserModelImpl implements TeaserModel {
    @ValueMapValue
    private String title;
}
```

#### @Cleanup — Resource Cleanup

`@Cleanup` calls `close()` at the end of the scope. Useful but overlaps with try-with-resources:

```java
// CORRECT: @Cleanup for ResourceResolver obtained from a factory
public void processAssets(ResourceResolverFactory resolverFactory) throws LoginException {
    @Cleanup ResourceResolver resolver = resolverFactory.getServiceResourceResolver(authInfo);
    Resource root = resolver.getResource("/content/dam/myproject");
    // resolver.close() is called automatically at end of method
}
```

```java
// PREFERRED: try-with-resources is more explicit and standard Java
public void processAssets(ResourceResolverFactory resolverFactory) throws LoginException {
    try (ResourceResolver resolver = resolverFactory.getServiceResourceResolver(authInfo)) {
        Resource root = resolver.getResource("/content/dam/myproject");
    }
}
```

**Recommendation**: Prefer try-with-resources for clarity. Use `@Cleanup` only when you have multiple resources and nesting becomes unwieldy.

#### @SneakyThrows — Selective Use Only

`@SneakyThrows` suppresses checked exceptions without declaring them. In AEM, use sparingly:

```java
// ACCEPTABLE: @SneakyThrows on a servlet where the framework already handles exceptions
@Slf4j
@Component(service = Servlet.class,
    property = {
        "sling.servlet.paths=/bin/myproject/export",
        "sling.servlet.methods=GET"
    })
public class ExportServlet extends SlingSafeMethodsServlet {

    @SneakyThrows(IOException.class)
    @Override
    protected void doGet(SlingHttpServletRequest request, SlingHttpServletResponse response) {
        response.setContentType("application/json");
        objectMapper.writeValue(response.getOutputStream(), buildResponse(request));
    }
}
```

```java
// INCORRECT: @SneakyThrows hiding repository exceptions — these MUST be handled
@SneakyThrows  // Hides RepositoryException — callers have no idea this can fail
public List<String> getTags(Session session) {
    QueryManager qm = session.getWorkspace().getQueryManager();
    // If this throws RepositoryException, it becomes an unchecked exception
    // and no caller will catch it
}
```

**Rule**: Only use `@SneakyThrows` when the exception is already handled by a framework layer (servlet container, workflow engine) or when you are absolutely certain the exception cannot occur in practice.

#### @Accessors(fluent = true) — Fluent APIs

```java
// CORRECT: Fluent accessors on a builder-like DTO
@Getter
@Setter
@Accessors(fluent = true)
public class QueryParams {
    private String rootPath;
    private String nodeType;
    private int limit;
    private int offset;
}

// Usage:
QueryParams params = new QueryParams()
    .rootPath("/content/mysite")
    .nodeType("cq:Page")
    .limit(10)
    .offset(0);
```

**Warning**: Fluent accessors drop the `get`/`set` prefix. This breaks HTL access (HTL calls `getTitle()`, not `title()`). Never use `@Accessors(fluent = true)` on Sling Models.

#### @Delegate — Interface Delegation

```java
// CORRECT: Delegating Core Component interface methods
@Model(adaptables = SlingHttpServletRequest.class,
       adapters = Teaser.class,
       resourceType = "myproject/components/teaser")
public class CustomTeaserImpl implements Teaser {

    @Self
    @Via(type = ResourceSuperType.class)
    @Delegate(excludes = DelegationExclusion.class)
    private Teaser delegate;

    // Override only what you need
    @Override
    public String getTitle() {
        return StringUtils.upperCase(delegate.getTitle());
    }

    private interface DelegationExclusion {
        String getTitle();
    }
}
```

### 3. Lombok with Sling Models

#### Standard Sling Model Pattern with Lombok

This is the recommended pattern for Sling Models using Lombok:

```java
// CORRECT: The gold standard Sling Model with Lombok
@Slf4j
@Getter
@Model(adaptables = SlingHttpServletRequest.class,
       adapters = ArticleComponent.class,
       resourceType = "myproject/components/article",
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class ArticleComponentImpl implements ArticleComponent {

    @ValueMapValue
    private String title;

    @ValueMapValue
    private String subtitle;

    @ValueMapValue
    private String author;

    @ValueMapValue
    private String imagePath;

    @ValueMapValue
    @Default(values = "Read More")
    private String ctaLabel;

    @ValueMapValue
    private String ctaLink;

    @ValueMapValue
    @Default(booleanValues = false)
    private boolean hideImage;

    @ChildResource
    private List<LinkItem> links;

    private String formattedDate;

    @PostConstruct
    private void init() {
        log.debug("Initializing ArticleComponent for resource: {}",
                  getResource().getPath());
        formattedDate = computeFormattedDate();
    }

    @SlingObject
    private Resource resource;

    private Resource getResource() {
        return resource;
    }

    private String computeFormattedDate() {
        // Complex date formatting logic
        return DateTimeFormatter.ISO_LOCAL_DATE.format(LocalDate.now());
    }
}
```

HTL accesses these as `${model.title}`, `${model.ctaLabel}`, `${model.links}`, etc. The `@Getter` at class level generates all getter methods.

#### Why NOT @Data on @Model Classes

`@Data` generates `@RequiredArgsConstructor`, which creates a constructor with parameters for all `final` fields. Sling's model injection framework instantiates models with a no-argument constructor (or a single-argument constructor taking the adaptable), then injects fields. The extra constructor confuses the injection framework.

```java
// INCORRECT: @Data on Sling Model
@Data
@Model(adaptables = Resource.class,
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class CardModelImpl implements CardModel {

    @ValueMapValue
    private String heading;

    @ValueMapValue
    private String description;

    // @Data generates: setters (unnecessary — Sling injects fields),
    // @RequiredArgsConstructor (no required final fields here, but pattern is dangerous),
    // @EqualsAndHashCode (wrong for JCR-backed object),
    // @ToString (may include Resource fields, causing issues)
}

// CORRECT: Use specific annotations
@Getter
@Slf4j
@ToString(exclude = "resource")
@Model(adaptables = Resource.class,
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class CardModelImpl implements CardModel {

    @ValueMapValue
    private String heading;

    @ValueMapValue
    private String description;

    @SlingObject
    private Resource resource;
}
```

#### Lombok with @PostConstruct

`@PostConstruct` works perfectly with Lombok. The lifecycle is: Sling instantiates the object, injects fields, then calls `@PostConstruct`. Lombok annotations do not interfere with this lifecycle because they only generate getters/setters/constructors — they do not affect field injection.

```java
@Slf4j
@Getter
@Model(adaptables = SlingHttpServletRequest.class,
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class NavigationModelImpl implements NavigationModel {

    @ValueMapValue
    private String rootPath;

    @ValueMapValue
    @Default(intValues = 3)
    private int depth;

    // This field is NOT injected — it is computed in @PostConstruct
    private List<NavItem> items;

    @OSGiService
    private PageManager pageManager;

    @PostConstruct
    private void init() {
        log.debug("Building navigation from root={} depth={}", rootPath, depth);
        items = buildNavTree(rootPath, depth);
    }

    private List<NavItem> buildNavTree(String path, int maxDepth) {
        // navigation building logic
        return Collections.emptyList();
    }
}
```

#### Lombok with @ChildResource and Collections

```java
@Getter
@Model(adaptables = Resource.class,
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class AccordionModelImpl implements AccordionModel {

    @ValueMapValue
    private String title;

    @ChildResource
    private List<AccordionItem> items;

    // The inner child model also benefits from Lombok
    @Getter
    @Model(adaptables = Resource.class,
           defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
    public static class AccordionItem {

        @ValueMapValue
        private String heading;

        @ValueMapValue
        private String content;

        @ValueMapValue
        @Default(booleanValues = false)
        private boolean expandedByDefault;
    }
}
```

#### @Builder.Default for Default Values in DTOs

When using `@Builder` alongside Sling Models (for building response DTOs, not on the model itself):

```java
@Builder
@Getter
public class ComponentConfig {
    private final String componentType;
    @Builder.Default
    private final boolean cacheable = true;
    @Builder.Default
    private final int ttl = 3600;
    @Builder.Default
    private final Map<String, String> properties = new HashMap<>();
}
```

### 4. Lombok with OSGi Services

#### @Slf4j for Service Logging

```java
// CORRECT: Standard OSGi service with Lombok logging
@Slf4j
@Component(service = EmailService.class, immediate = true)
public class EmailServiceImpl implements EmailService {

    @Reference
    private MessageGatewayService messageGatewayService;

    @Reference
    private ResourceResolverFactory resolverFactory;

    @Override
    public boolean sendEmail(String templatePath, Map<String, String> params, String... recipients) {
        log.info("Sending email using template: {} to {} recipients", templatePath, recipients.length);
        try {
            // email sending logic
            return true;
        } catch (Exception e) {
            log.error("Failed to send email using template: {}", templatePath, e);
            return false;
        }
    }
}
```

#### @RequiredArgsConstructor Does NOT Work with OSGi

OSGi Declarative Services (DS) uses field injection via `@Reference`. It instantiates the component with a no-arg constructor, then injects references. `@RequiredArgsConstructor` generates a constructor that OSGi DS cannot use.

```java
// INCORRECT: @RequiredArgsConstructor with @Reference — OSGi cannot use this constructor
@Slf4j
@RequiredArgsConstructor
@Component(service = PageService.class)
public class PageServiceImpl implements PageService {

    private final ResourceResolverFactory resolverFactory;  // OSGi cannot inject via constructor
    private final Externalizer externalizer;                 // OSGi uses field injection, not constructor

    // This will fail at runtime — OSGi DS requires a no-arg constructor
}
```

```java
// CORRECT: Let OSGi handle injection — no constructor annotations
@Slf4j
@Component(service = PageService.class)
public class PageServiceImpl implements PageService {

    @Reference
    private ResourceResolverFactory resolverFactory;

    @Reference
    private Externalizer externalizer;

    @Override
    public String getExternalLink(String path) {
        log.debug("Externalizing path: {}", path);
        return externalizer.publishLink(resolverFactory, path);
    }
}
```

#### @Getter(lazy = true) for Expensive Computations

```java
@Slf4j
@Component(service = ConfigurationService.class)
public class ConfigurationServiceImpl implements ConfigurationService {

    @Reference
    private ResourceResolverFactory resolverFactory;

    // Computed lazily on first access, then cached
    @Getter(lazy = true)
    private final Map<String, String> siteConfigMap = loadSiteConfig();

    private Map<String, String> loadSiteConfig() {
        log.info("Loading site configuration — this is expensive and will be cached");
        Map<String, String> config = new HashMap<>();
        // ... load from JCR
        return Collections.unmodifiableMap(config);
    }
}
```

**Note**: `@Getter(lazy = true)` requires the field to be `private final` and initialized inline. The value is computed once on first access and cached in a thread-safe manner (using `AtomicReference` internally).

### 5. Lombok with Servlets and Filters

#### @Slf4j on Sling Servlets

```java
@Slf4j
@Component(service = Servlet.class,
    property = {
        "sling.servlet.paths=/bin/myproject/search",
        "sling.servlet.methods=GET"
    })
public class SearchServlet extends SlingSafeMethodsServlet {

    @Reference
    private SearchService searchService;

    @Override
    protected void doGet(SlingHttpServletRequest request, SlingHttpServletResponse response)
            throws ServletException, IOException {
        String query = request.getParameter("q");
        log.info("Search request received: q={}", query);

        List<SearchResultDto> results = searchService.search(query);

        response.setContentType("application/json");
        response.setCharacterEncoding("UTF-8");
        new ObjectMapper().writeValue(response.getWriter(), results);
    }
}
```

#### DTO Classes with @Data for JSON Serialization

Create dedicated DTO classes for servlet JSON responses:

```java
// Response DTO — @Data is appropriate here (not a Sling Model)
@Data
public class SearchResultDto {
    private String title;
    private String path;
    private String excerpt;
    private double score;
    private String thumbnail;
}
```

```java
// Response wrapper with @Builder
@Builder
@Getter
public class ApiResponse<T> {
    private final int status;
    private final String message;
    private final T data;
    @Builder.Default
    private final long timestamp = System.currentTimeMillis();
}

// Usage in servlet:
ApiResponse<List<SearchResultDto>> response = ApiResponse.<List<SearchResultDto>>builder()
    .status(200)
    .message("OK")
    .data(results)
    .build();
```

#### @SneakyThrows on doGet/doPost — Use With Caution

```java
// ACCEPTABLE: When the servlet container already handles these exceptions
@Slf4j
@Component(service = Servlet.class,
    property = {
        "sling.servlet.paths=/bin/myproject/export",
        "sling.servlet.methods=POST"
    })
public class DataExportServlet extends SlingAllMethodsServlet {

    @Reference
    private ExportService exportService;

    @SneakyThrows({IOException.class})
    @Override
    protected void doPost(SlingHttpServletRequest request, SlingHttpServletResponse response) {
        // IOException is declared by the parent method signature anyway —
        // @SneakyThrows here just avoids re-declaring it when overriding
        String json = IOUtils.toString(request.getReader());
        ExportRequest exportRequest = new ObjectMapper().readValue(json, ExportRequest.class);
        byte[] data = exportService.export(exportRequest);
        response.getOutputStream().write(data);
    }
}
```

**Note**: `doGet` and `doPost` already declare `throws ServletException, IOException`. Using `@SneakyThrows` for these specific exceptions adds no real value. It is more useful when calling methods that throw checked exceptions not declared in the servlet method signature.

### 6. Lombok with Jackson JSON Serialization

#### @Data + @JsonInclude for Sling Model Exporters

When using the Sling Model Exporter framework (`.model.json`), Lombok-generated getters are picked up by Jackson:

```java
@Getter
@Slf4j
@Model(adaptables = SlingHttpServletRequest.class,
       adapters = { ProductCard.class, ComponentExporter.class },
       resourceType = "myproject/components/product-card",
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
@Exporter(name = ExporterConstants.SLING_MODEL_EXPORTER_NAME,
          extensions = ExporterConstants.SLING_MODEL_EXTENSION)
@JsonInclude(JsonInclude.Include.NON_NULL)
public class ProductCardImpl implements ProductCard, ComponentExporter {

    @ValueMapValue
    private String productName;

    @ValueMapValue
    private String sku;

    @ValueMapValue
    private Double price;

    @ValueMapValue
    private String imagePath;

    @Override
    public String getExportedType() {
        return "myproject/components/product-card";
    }
}
```

The `@Getter` generates `getProductName()`, `getSku()`, etc. Jackson serializes these as `{"productName": "...", "sku": "...", "price": ...}`.

#### @Builder + @JsonDeserialize for Immutable JSON DTOs

```java
@Value
@Builder
@JsonDeserialize(builder = CreatePageRequest.CreatePageRequestBuilder.class)
public class CreatePageRequest {
    String parentPath;
    String pageName;
    String template;
    Map<String, String> properties;

    @JsonPOJOBuilder(withPrefix = "")
    public static class CreatePageRequestBuilder {
        // Lombok generates the builder methods — Jackson uses this class for deserialization
    }
}
```

#### @JsonProperty with Lombok Getters — Naming Conflicts

Lombok generates getter names following JavaBean conventions (`getXUrl()` for field `xUrl`). Jackson derives the JSON property name from the getter. This can cause unexpected JSON keys.

```java
// PROBLEM: Field "xUrl" generates getter "getXUrl()" — Jackson serializes as "xurl" (lowercased)
@Getter
public class LinkDto {
    private String xUrl;  // JSON output: {"xurl": "..."} — unexpected!
}

// FIX: Use @JsonProperty to enforce the correct name
@Getter
public class LinkDto {
    @JsonProperty("xUrl")
    private String xUrl;  // JSON output: {"xUrl": "..."} — correct
}
```

```java
// PROBLEM: Boolean field "isActive" generates getter "isActive()" — Jackson may serialize as "active"
@Getter
public class StatusDto {
    private boolean isActive;  // getter: isIsActive() — wrong!
}

// FIX: Name the field without the "is" prefix
@Getter
public class StatusDto {
    private boolean active;  // getter: isActive() — Jackson serializes as "active" — correct
}
```

### 7. Common Pitfalls and Anti-Patterns

#### @Data on Sling Models Breaks Injection

This is the single most common Lombok mistake in AEM projects:

```java
// BROKEN: @Data generates @RequiredArgsConstructor, @Setter, @EqualsAndHashCode, @ToString
// The constructors and equals/hashCode conflict with Sling's injection and Resource identity
@Data
@Model(adaptables = Resource.class,
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class HeroModelImpl implements HeroModel {
    @ValueMapValue
    private String title;

    @ValueMapValue
    private String subtitle;
}

// CORRECT: Use only the annotations you need
@Getter
@Slf4j
@Model(adaptables = Resource.class,
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class HeroModelImpl implements HeroModel {
    @ValueMapValue
    private String title;

    @ValueMapValue
    private String subtitle;
}
```

#### @EqualsAndHashCode on JCR-Backed Objects

JCR Resources and Nodes use identity equality (same path = same resource). Generating value-based `equals` and `hashCode` on objects wrapping Resources leads to broken Set/Map behavior:

```java
// INCORRECT: Value-based equality on a JCR-backed object
@EqualsAndHashCode
@Getter
@Model(adaptables = Resource.class,
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class PageItemImpl implements PageItem {
    @ValueMapValue
    private String title;

    @SlingObject
    private Resource resource;
    // Two different resources with the same title would be considered "equal" — wrong
}

// CORRECT: If you need equality, base it on the resource path
@Getter
@Model(adaptables = Resource.class,
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class PageItemImpl implements PageItem {
    @ValueMapValue
    private String title;

    @SlingObject
    private Resource resource;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof PageItemImpl)) return false;
        return Objects.equals(resource.getPath(), ((PageItemImpl) o).resource.getPath());
    }

    @Override
    public int hashCode() {
        return Objects.hash(resource.getPath());
    }
}
```

#### @ToString Including Resource/Node Fields

`Resource.toString()` and `Node.toString()` can produce enormous output or trigger lazy loading of JCR data. Always exclude them:

```java
// INCORRECT: Includes Resource in toString — can dump entire subtree
@ToString
@Model(adaptables = Resource.class)
public class ContentBlockImpl implements ContentBlock {
    @SlingObject
    private Resource resource;
    @ValueMapValue
    private String text;
}

// CORRECT: Exclude non-serializable fields
@ToString(exclude = {"resource", "request", "resolver"})
@Model(adaptables = SlingHttpServletRequest.class)
public class ContentBlockImpl implements ContentBlock {
    @Self
    private SlingHttpServletRequest request;
    @SlingObject
    private Resource resource;
    @ValueMapValue
    private String text;
}

// ALTERNATIVE: Include only the fields you want
@ToString(onlyExplicitlyIncluded = true)
@Model(adaptables = Resource.class)
public class ContentBlockImpl implements ContentBlock {
    @SlingObject
    private Resource resource;

    @ToString.Include
    @ValueMapValue
    private String text;
}
```

#### Lombok + AEM Mocks Compatibility

Lombok-annotated classes work seamlessly with AEM Mocks (io.wcm). Lombok processes at compile time, so by the time tests run, the bytecode already has all generated methods:

```java
@ExtendWith(AemContextExtension.class)
class ArticleComponentImplTest {

    private final AemContext context = new AemContext();

    @Test
    void testGetTitle() {
        context.build().resource("/content/test",
            "title", "Test Article",
            "sling:resourceType", "myproject/components/article");
        context.currentResource("/content/test");

        // Lombok's @Getter generates getTitle() — AEM Mocks calls it normally
        ArticleComponent model = context.request().adaptTo(ArticleComponent.class);
        assertEquals("Test Article", model.getTitle());
    }
}
```

No special test configuration is needed for Lombok.

#### @Cleanup with try-with-resources Redundancy

Do not combine `@Cleanup` with try-with-resources — it causes double-close:

```java
// INCORRECT: Double-close — both @Cleanup and try-with-resources will call close()
try (@Cleanup ResourceResolver resolver = resolverFactory.getServiceResourceResolver(authInfo)) {
    // resolver.close() will be called TWICE
}

// CORRECT: Use one or the other
// Option A: try-with-resources (preferred — standard Java idiom)
try (ResourceResolver resolver = resolverFactory.getServiceResourceResolver(authInfo)) {
    // ...
}

// Option B: @Cleanup (when nesting becomes deep)
@Cleanup ResourceResolver resolver = resolverFactory.getServiceResourceResolver(authInfo);
// ...
```

#### Over-Using @SneakyThrows

```java
// INCORRECT: Hiding RepositoryException — callers cannot handle JCR failures
@SneakyThrows
public List<String> getChildPages(Session session, String parentPath) {
    Node parent = session.getNode(parentPath);
    // If the node doesn't exist, RepositoryException is thrown as unchecked —
    // no caller will ever know to catch it
}

// CORRECT: Declare or handle the exception
public List<String> getChildPages(Session session, String parentPath) throws RepositoryException {
    Node parent = session.getNode(parentPath);
    // Caller can catch RepositoryException and handle gracefully
}

// ALSO CORRECT: Catch and handle within the method
public List<String> getChildPages(Session session, String parentPath) {
    try {
        Node parent = session.getNode(parentPath);
        // ...
        return results;
    } catch (RepositoryException e) {
        log.error("Failed to read child pages under: {}", parentPath, e);
        return Collections.emptyList();
    }
}
```

### 8. Configuration

#### lombok.config File

Place a `lombok.config` file at the root of each Maven module (e.g., `core/lombok.config`) to enforce project-wide Lombok settings:

```properties
# core/lombok.config — Recommended AEM project configuration

# Stop Lombok from searching parent directories for more config files
config.stopBubbling = true

# Use SLF4J for @Log (enforce correct AEM logging)
lombok.log.fieldName = log
lombok.log.fieldIsStatic = true

# Suppress @ToString on types extending Resource or Node — prevent accidental serialization
lombok.toString.doNotUseGetters = true

# Add @Generated annotation to Lombok-generated code (helps code coverage tools)
lombok.addLombokGeneratedAnnotation = true

# Flag @Data usage — warn developers to use @Getter + specific annotations instead
lombok.data.flagUsage = WARNING

# Flag @Value usage with a reminder
lombok.value.flagUsage = WARNING

# Completely disallow experimental features in production code
lombok.experimental.flagUsage = WARNING

# Disallow @SneakyThrows to force explicit exception handling (optional — strict projects)
# lombok.sneakyThrows.flagUsage = WARNING

# Disallow @Cleanup to enforce try-with-resources (optional — strict projects)
# lombok.cleanup.flagUsage = WARNING
```

#### Flag Levels

| Level | Meaning |
|-------|---------|
| `WARNING` | Compiler warning — code still compiles |
| `ERROR` | Compiler error — code will not compile |
| `ALLOW` | No restriction (default for most annotations) |

#### Suppressing Warnings

To suppress a Lombok-related warning on a specific class:

```java
@SuppressWarnings("lombok")
@Data  // Normally flagged by lombok.data.flagUsage = WARNING
public class LegacyDto {
    private String name;
    private String value;
}
```

#### Recommended Project Structure for lombok.config

```
myproject/
  core/
    lombok.config          <-- AEM bundle module config
    src/main/java/...
  it.tests/
    lombok.config          <-- Test module can be more permissive
    src/main/java/...
```

The `it.tests/lombok.config` can be less strict:

```properties
config.stopBubbling = true
lombok.addLombokGeneratedAnnotation = true
# Allow @Data in test code — test DTOs benefit from it
lombok.data.flagUsage = ALLOW
lombok.experimental.flagUsage = ALLOW
```

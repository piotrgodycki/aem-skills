## ClientLibs Configuration & Best Practices

Client-Side Libraries (ClientLibs) are AEM's mechanism for organizing and delivering CSS and JavaScript. On AEM Cloud Service, proper configuration is critical for publish access and CDN caching.

### Folder Structure

```
/apps/myproject/clientlibs/
├── clientlib-base/
│   ├── .content.xml        # cq:ClientLibraryFolder node
│   ├── css/
│   │   └── styles.css
│   ├── js/
│   │   └── scripts.js
│   ├── resources/           # Static assets (fonts, icons)
│   ├── css.txt              # CSS manifest
│   └── js.txt               # JS manifest
├── clientlib-site/          # Generated by ui.frontend
├── clientlib-dependencies/  # Generated by ui.frontend
└── clientlib-author/        # Author-only customizations
```

### Node Properties (.content.xml)

```xml
<?xml version="1.0" encoding="UTF-8"?>
<jcr:root xmlns:cq="http://www.day.com/jcr/cq/1.0"
    xmlns:jcr="http://www.jcp.org/jcr/1.0"
    jcr:primaryType="cq:ClientLibraryFolder"
    allowProxy="{Boolean}true"
    categories="[myproject.base]"
    dependencies="[myproject.dependencies]"
    embed="[myproject.utils]"/>
```

| Property | Type | Purpose |
|----------|------|---------|
| `categories` | String[] | Library identifier for inclusion |
| `allowProxy` | Boolean | **REQUIRED for Cloud Service** — serves via `/etc.clientlibs/` |
| `dependencies` | String[] | Libraries that must load first (separate HTTP requests) |
| `embed` | String[] | Libraries to inline into this one (reduces HTTP requests) |

### Manifest Files (css.txt / js.txt)

```
#base=css
styles.css
layout.css
```

The `#base` directive sets the root folder. Files are concatenated in listed order.

### Proxy Servlet Pattern (Cloud Service)

With `allowProxy=true`, libraries under `/apps` become accessible at `/etc.clientlibs/`:

```
Library: /apps/myproject/clientlibs/clientlib-base
Access:  /etc.clientlibs/myproject/clientlibs/clientlib-base.css
Assets:  /etc.clientlibs/myproject/clientlibs/clientlib-base/resources/icon.png
```

### HTL/Sightly Integration

```html
<!-- Load all (CSS + JS) -->
<sly data-sly-use.clientLib="${'/libs/granite/sightly/templates/clientlib.html'}"
     data-sly-call="${clientLib.all @ categories='myproject.base'}"/>

<!-- CSS only (in <head>) -->
<sly data-sly-call="${clientLib.css @ categories='myproject.base'}"/>

<!-- JS only (before </body>) -->
<sly data-sly-call="${clientLib.js @ categories='myproject.base'}"/>

<!-- Multiple categories -->
<sly data-sly-call="${clientLib.all @ categories='myproject.base,myproject.widgets'}"/>
```

### Dependencies vs Embedding

- **Dependencies**: Separate HTTP requests, loaded before this library. Use for shared libraries needed by multiple ClientLibs
- **Embedding**: Inlined into a single file, reducing HTTP requests. Use for libraries only needed by this ClientLib

### Versioned URLs (Immutable Caching)

```
/etc.clientlibs/myproject/clientlibs/clientlib-base.lc-7c8c5d228445ff48ab49a8e3c865c562-lc.css
```

Hash changes when content changes, enabling immutable/long-term caching.

### Debugging

- `?debugClientLibs=true` — Shows individual files instead of concatenated bundles
- `/libs/granite/ui/content/dumplibs.test.html` — Lists all ClientLib folders

**Incorrect — missing proxy, wrong location:**

```xml
<!-- Missing allowProxy — inaccessible on publish -->
<jcr:root jcr:primaryType="cq:ClientLibraryFolder"
    categories="[myproject.base]"/>

<!-- Under /etc — not allowed on Cloud Service -->
/etc/clientlibs/myproject/...
```

**Correct:**

```xml
<!-- allowProxy enabled, under /apps -->
<jcr:root jcr:primaryType="cq:ClientLibraryFolder"
    allowProxy="{Boolean}true"
    categories="[myproject.base]"/>
```

### Pitfalls

- Missing `allowProxy` makes libraries inaccessible on publish
- Circular dependencies between ClientLibs cause load failures
- Embedding large libraries inflates bundle size — prefer dependencies for shared code
- ClientLibs MUST be under `/apps` (not `/etc`) on Cloud Service
- Static resources outside `resources/` subfolder won't be accessible on publish
- Don't manually minify — use preprocessor options (`cssProcessor`, `jsProcessor`)
